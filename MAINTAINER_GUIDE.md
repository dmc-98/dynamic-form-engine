# Maintainer Guide

This guide is the practical operating manual for shipping and growing Dynamic Form Engine under the `@dmc-98` package scope.

## Release Basics

Start here for the first public release:

1. Reserve the npm scope `@dmc-98`.
2. Add `NPM_TOKEN` to GitHub repository secrets.
3. Keep `main` green.
4. Run the verification list in `RELEASE.md`.
5. Merge PRs with changesets.
6. Review and merge the automated release PR.

Use these commands from the repo root:

```bash
pnpm install
pnpm build
pnpm test
pnpm typecheck
pnpm test:coverage:stable
pnpm test:smoke:artifacts
pnpm test:smoke:wrappers
pnpm release:check
pnpm --dir docs build
```

## Recommended GitHub Automations And Bots

Keep the automation stack lean and high-signal early on:

- Dependabot: security and dependency update PRs.
- CodeQL: security scanning for the monorepo.
- Changesets release workflow: versioning and publish PRs.
- GitHub autogenerated release notes: fast baseline release notes for each tag.
- `actions/first-interaction`: welcome first-time contributors.
- `allcontributors` bot: optional, but useful once outside contributors start landing.

Avoid early:

- Aggressive stale bots that auto-close issues.
- Multiple competing release-note tools on top of Changesets.

## Codex Automation Ideas

These work well as Codex recurring tasks:

- PR review automation: check open PRs, summarize risks, and flag release blockers.
- Weekly feature implementation automation: inspect requested features, pick a scoped winner, implement it on a branch, verify, and open a PR summary.
- Docs freshness automation: scan `README`, release guides, and docs nav for stale package names or broken references.

The suggested automation directives are returned in the assistant response when you ask for them, so they can be created from the UI without hand-editing files.

## Release Notes Checklist

Every public release should include:

- One paragraph of highlights.
- Stable, Beta, and Experimental status changes.
- New features.
- Bug fixes.
- Upgrade or migration notes.
- Docs and example changes.
- Known limitations or deliberate non-goals.
- Contributor thanks.
- Ecosystem acknowledgements where relevant.

## Marketing Plan

Good open-source launches are repetitive, not one-shot.

### Before Launch

- Clean up the README and docs landing pages.
- Record a short GIF or screen capture of the Playground and canonical example.
- Prepare a one-paragraph positioning statement: what DFE does and who it is for.
- Prepare a short comparison page so users understand where DFE fits.

### Launch Week

- Publish a GitHub Release with complete notes.
- Post a launch thread on X and LinkedIn.
- Share in GitHub Discussions.
- Post in a few targeted communities, not everywhere at once.
- Ask for feedback on one or two sharp questions, not generic “thoughts?”

### Ongoing

- Publish a changelog for every release.
- Share concrete examples, benchmarks, and adoption stories.
- Write comparison content honestly and respectfully.
- Open GitHub Sponsors once the README, docs, and release flow feel polished.

## Sponsorship Readiness

Before actively seeking sponsorship:

- Make sure the stable package lane is clearly documented.
- Keep issues and PRs responsive.
- Publish consistent releases.
- Add a sponsors section to the README and GitHub profile.
- Prepare a sponsor pitch: what support unlocks next.

Good early sponsorship messaging:

- fund ecosystem maintenance
- sponsor serverless adapters and examples
- sponsor visual builder and enterprise hardening
- sponsor docs, templates, and integrations

## Docs Upkeep

Before each release:

- Verify the package scope is `@dmc-98` everywhere.
- Rebuild the docs locally.
- Update comparison content if product positioning changed.
- Refresh acknowledgements when major new dependencies or contributors land.
- Check the Playground docs and launch paths.

## Playground Deployment Recommendation

Use this split:

- GitHub Pages for docs.
- Cloudflare Pages for the public Playground.
- A real serverless runtime for collaboration backends and example APIs.

Why:

- GitHub Pages is static hosting only and works best for docs or a static SPA preview.
- Cloudflare Pages handles static frontends well and leaves room for Pages Functions later if the Playground grows server-side features.
- The canonical collaboration and backend examples need a real serverless or Node runtime and should not be marketed as GitHub Pages deploys.

## Scope And Brand Changes

If the project later moves the GitHub org or site URLs:

1. Update `dfe.config.ts`.
2. Run `pnpm run sync-config`.
3. Verify package names, docs links, badges, examples, and workflows.
4. Rebuild docs and rerun release verification.
